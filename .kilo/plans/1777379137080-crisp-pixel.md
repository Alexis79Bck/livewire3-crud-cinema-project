## Phase 1: Foundation & Database (Weeks 1-2)

### Sprint 1.1: Database Schema & Migrations

**Objective**: Create all required database tables with proper relationships and indexes

**Tasks**:
1. **Create Auditoriums Migration**
   - Table: `auditoriums`
   - Fields: id, name, capacity, is_active, timestamps
   - Indexes: `idx_is_active`
   - Reference: Theater context aggregate root

2. **Create Seats Migration**
   - Table: `seats`
   - Fields: id, auditorium_id, row, number, type (enum: STANDARD, VIP, ACCESSIBLE), position_x, position_y
   - Foreign Keys: `seats.auditorium_id → auditoriums.id` (CASCADE delete)
   - Indexes: `idx_auditorium`, `idx_seat_location` composite (auditorium_id, row, number)
   - Unique constraint: (auditorium_id, row, number)

3. **Create Showtimes Migration**
   - Table: `showtimes`
   - Fields: id, movie_id, auditorium_id, start_time, end_time, base_price, is_active
   - Foreign Keys:
     - `showtimes.movie_id → movies.id` (RESTRICT)
     - `showtimes.auditorium_id → auditoriums.id` (RESTRICT)
   - Indexes: `idx_movie`, `idx_auditorium`, `idx_start_time`, composite `idx_movie_start`
   - Business Rule: end_time > start_time

4. **Create Bookings Migration**
   - Table: `bookings`
   - Fields: id, user_id, showtime_id, booking_status (enum), total_amount, currency, expires_at, confirmed_at, cancelled_at, metadata (JSON)
   - Foreign Keys: `bookings.showtime_id → showtimes.id` (CASCADE)
   - Indexes: `idx_user`, `idx_showtime`, `idx_status`, `idx_expires_at`
   - Unique constraint per active booking per user+showtime

5. **Create Tickets Migration**
   - Table: `tickets`
   - Fields: id, booking_id, seat_id, showtime_id, ticket_status (enum), price, currency, qr_code, checksum
   - Foreign Keys:
     - `tickets.booking_id → bookings.id` (CASCADE)
     - `tickets.seat_id → seats.id` (RESTRICT)
     - `tickets.showtime_id → showtimes.id` (RESTRICT)
   - Indexes: `idx_booking`, `idx_seat_showtime` composite (seat_id, showtime_id), `idx_qr_code`
   - Unique constraint: (seat_id, showtime_id) for active tickets

6. **Add Enums to Database**
   - Create migrations for: `seat_types`, `booking_statuses`, `ticket_statuses`
   - Or use string columns with ENUM types in migration

**Deliverables**:
- 5 migration files in `database/migrations/`
- Run `php artisan migrate` to apply
- Verify with `php artisan migrate:status`

### Sprint 1.2: Domain Layer - Core Aggregates

**Objective**: Implement Booking, Showtime, and Auditorium aggregates with full domain logic

**Pattern Reference**: `app/Domain/Catalog/Aggregates/Movie/Movie.php` (complete reference)

#### 1. Booking Aggregate (Root Entity)

**Files to create**:
- `app/Domain/Booking/Aggregates/Booking/Booking.php` (main aggregate)
- `app/Domain/Booking/Aggregates/Booking/BookingId.php` (value object)
- `app/Domain/Booking/Aggregates/Booking/Customer.php` (value object)
- `app/Domain/Booking/Aggregates/Booking/Ticket.php` (entity)
- `app/Domain/Booking/Aggregates/Booking/BookingStatus.php` (enum/value object)

**Domain Logic**:
- State Machine: PENDING → CONFIRMED / CANCELLED / EXPIRED / COMPLETED
- Invariants:
  - Customer cannot book same seat twice in same showtime
  - Total amount = sum(ticket prices) + fees
  - Booking expires after configurable time (default 15 min)
  - Cannot cancel confirmed booking within cancellation window (e.g., 2h before show)
  - Cannot exceed auditorium capacity
- Methods:
  - `createFromCart()` - factory method
  - `addTicket()` - add ticket with seat validation
  - `confirm()` - transition to confirmed, generate ticket QR codes
  - `cancel()` - transition to cancelled, release seats
  - `expire()` - transition to expired, release seats
  - `calculateTotal()` - aggregate ticket prices + fees
  - `getAvailableSeats()` - query available seats for showtime
  - `isCancellable()` - business rule check
- Domain Events:
  - `BookingCreated`
  - `BookingConfirmed`
  - `BookingCancelled`
  - `BookingExpired`
  - `TicketIssued`

#### 2. Showtime Aggregate (Root Entity)

**Domain Logic**:

## Phase 3: Application Layer Handlers (Week 3-4)

**Pattern**: Follow `app/Application/Catalog/Movie/Handlers/` structure

### Booking Commands & Handlers:

1. **CreateBookingCommand** → `CreateBookingHandler.php`
   - Input: userId, showtimeId, seatIds, customer info
   - Output: BookingId
   - Steps: Validate → Lock Seats → Calculate Price → Create Booking → Release lock on failure

2. **ConfirmBookingCommand** → `ConfirmBookingHandler.php`
   - Input: bookingId, paymentMethod
   - Output: Confirmation
   - Steps: Validate booking → Process payment → Issue tickets → Release locks

3. **CancelBookingCommand** → `CancelBookingHandler.php`
   - Input: bookingId, reason
   - Output: CancellationConfirmation
   - Steps: Validate cancellable → Refund if paid → Release seats → Update status

4. **ReserveSeatsCommand** → `ReserveSeatsHandler.php`
   - Input: showtimeId, seatIds, duration (optional)
   - Output: ReservationId with expiry
   - Steps: Lock seats → Create pending reservation

### Scheduling Commands & Handlers:

1. **CreateShowtimeCommand** → `CreateShowtimeHandler.php`
   - Input: movieId, auditoriumId, startTime, endTime
   - Validation: Auditorium available, no conflicts, movie active

2. **UpdateShowtimeCommand** → `UpdateShowtimeHandler.php`
   - Input: showtimeId, new times
   - Validation: No overlapping, not started, no bookings

3. **CancelShowtimeCommand** → `CancelShowtimeHandler.php`
   - Cascade cancellation of affected bookings

### Theater Commands & Handlers:

1. **CreateAuditoriumCommand** → `CreateAuditoriumHandler.php`
2. **UpdateSeatMapCommand** → `UpdateSeatMapHandler.php`

**All handlers**:
- Return DTOs or aggregate IDs
- Throw domain exceptions for business rule violations
- Dispatch domain events
- Transactional boundaries

```php
// Booking routes (authenticated)
Route::middleware('auth:sanctum')->group(function () {
    Route::prefix('bookings')->group(function () {
        Route::get('/', [BookingController::class, 'index']);
        Route::post('/', [BookingController::class, 'store']);
        Route::get('{booking}', [BookingController::class, 'show']);
        Route::post('{booking}/confirm', [BookingController::class, 'confirm']);
        Route::post('{booking}/cancel', [BookingController::class, 'cancel']);
        Route::get('showtimes/{showtime}/seats', [BookingController::class, 'getAvailableSeats']);
    });
});

// Showtime routes (public + admin)
Route::get('showtimes', [ShowtimeController::class, 'index']);
Route::middleware('auth:sanctum', 'role:admin')->group(function () {
    Route::post('showtimes', [ShowtimeController::class, 'store']);
    Route::put('showtimes/{showtime}', [ShowtimeController::class, 'update']);
    Route::delete('showtimes/{showtime}', [ShowtimeController::class, 'destroy']);
});

// Auditorium routes (admin only)
Route::middleware('auth:sanctum', 'role:admin')->group(function () {
    Route::apiResource('auditoriums', AuditoriumController::class)->only(['index', 'store', 'show', 'update', 'destroy']);
    Route::get('auditoriums/{auditorium}/seats', [AuditoriumController::class, 'getSeatMap']);
});
```

**Request Validation**:
- Use Laravel Form Requests (e.g., `CreateBookingRequest.php`)
- Validation rules mirror domain invariants

**Response Format**:
- JSON following API resources pattern
- Consistent envelope: `{ "data": ..., "message": ..., "errors": ... }`

## Implementation Order & Dependencies

### Critical Path:
```
Migrations → Domain Aggregates → Mappers → Repositories → Handlers → Controllers → Routes → Tests
```

### Dependencies:
- **Booking** depends on: Showtime (for showtime validation), Auditorium/Seat (for seat availability)
- **Showtime** depends on: Movie (Catalog - complete), Auditorium
- **Auditorium** independent (no dependencies)
- **Handlers** depend on: Aggregates + Repositories + Services
- **Controllers** depend on: Handlers + Request validation
- **Pricing** depends on: SeatType, Showtime time, Movie metadata

### Risk Mitigation:
1. **Seat Concurrency**: Use Redis locks + DB transactions for race conditions
2. **Payment Idempotency**: Unique payment intent IDs, handle duplicates
3. **Time-based Rules**: Use consistent timezone (UTC)
4. **Data Integrity**: Foreign key constraints + application-level validation
5. **Performance**: Indexes on all foreign keys + status + timestamp columns

## Technical Specifications

### Database Indexes (All Required):
```sql
-- Bookings
CREATE INDEX idx_bookings_user ON bookings(user_id);
CREATE INDEX idx_bookings_showtime ON bookings(showtime_id);
CREATE INDEX idx_bookings_status ON bookings(booking_status);
CREATE INDEX idx_bookings_expires ON bookings(expires_at) WHERE status = 'PENDING';

-- Tickets
CREATE INDEX idx_tickets_booking ON tickets(booking_id);
CREATE UNIQUE INDEX idx_tickets_seat_showtime ON tickets(seat_id, showtime_id) WHERE ticket_status != 'CANCELLED';

CREATE INDEX idx_showtimes_movie ON showtimes(movie_id);
CREATE INDEX idx_showtimes_auditorium ON showtimes(auditorium_id);
CREATE INDEX idx_showtimes_start ON showtimes(start_time);
CREATE INDEX idx_showtimes_movie_start ON showtimes(movie_id, start_time);

-- Seats
CREATE INDEX idx_seats_auditorium ON seats(auditorium_id);
CREATE UNIQUE INDEX idx_seats_unique ON seats(auditorium_id, row, number);
```
